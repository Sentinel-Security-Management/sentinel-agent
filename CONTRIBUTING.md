## Project Philosophy

Sentinel Agent is built in public.

We believe in:

* Transparent development
* Security through auditability
* Long-term sustainability
* High engineering standards
* Community collaboration

The project uses a release-based source-available licensing model that balances public access with sustainable development.


# Contributing to Sentinel Agent

Thank you for your interest in contributing to Sentinel Agent.

We welcome contributions from developers, security researchers, students, and infrastructure engineers.

## Before You Start

Please:

* Read the project documentation.
* Search existing issues before creating a new one.
* Open an issue for large changes before investing significant development effort.
* Ensure your changes align with the project's goals and architecture.

## Development Workflow

1. Fork the repository.
2. Create a feature branch.

```bash
git checkout -b feature/my-feature
```

3. Make your changes.
4. Add or update tests.
5. Ensure all tests pass.
6. Commit using clear commit messages.
7. Open a Pull Request.

## Pull Request Guidelines

A Pull Request should:

* Focus on a single logical change.
* Include documentation updates when applicable.
* Include tests when applicable.
* Explain the motivation behind the change.
* Pass all automated checks.

Large Pull Requests may be asked to be split into smaller reviewable units.

## Code Standards

* Write readable, maintainable code.
* Prefer simplicity over cleverness.
* Follow existing project conventions.
* Avoid introducing unnecessary dependencies.
* Document public interfaces.

## Security Issues

Do not disclose security vulnerabilities publicly.

Instead, report them privately to the maintainers.

Security reports should include:

* Affected component
* Reproduction steps
* Impact assessment
* Suggested remediation (if available)

* Distribute the contribution under the project's current license (FSL-1.1-ALv2).

* Distribute the contribution under Apache License 2.0 when the applicable release reaches its conversion date.

### Release-Based Conversion Policy

Sentinel follows a release-based licensing model.

Each tagged release is published under FSL-1.1-ALv2 and automatically converts to Apache License 2.0 twenty-four (24) months after its release date.

Contributors acknowledge that their contributions may become part of future releases and will follow the licensing terms applicable to those releases.

## Code of Conduct

Be respectful and professional.

Constructive criticism is encouraged.
Harassment, abuse, discrimination, and disruptive behavior are not tolerated.

Sentinel values meaningful contributions from the community.

At the discretion of the maintainers, contributors may receive rewards based on the quality, impact, and long-term value of their contributions to the project.

Potential rewards may include:

* Early access to new features
* Public recognition in release notes
* Contributor badges and acknowledgements
