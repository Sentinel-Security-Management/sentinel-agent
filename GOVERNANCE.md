# Sentinel Governance

## Purpose

This document defines how decisions are made within the Sentinel project and how contributors can participate in its development.

The goal of Sentinel governance is to ensure:

* High engineering standards
* Long-term sustainability
* Transparent development
* Consistent project direction
* Security-first decision making

---

## Governance Model

Sentinel currently follows a Founder-Led Open Development model.

The project is developed in public and welcomes community contributions, while strategic direction remains the responsibility of The Deployment Lab.

Community feedback is encouraged and valued, but final decisions remain with the project maintainers.

---

## Roles

### Founder

The Founder is responsible for:

* Project vision
* Architecture direction
* Licensing decisions
* Release strategy
* Maintainer appointments
* Governance updates

The Founder has final decision-making authority.

---

### Maintainers

Maintainers are trusted contributors appointed by The Deployment Lab.

Responsibilities include:

* Reviewing Pull Requests
* Triaging issues
* Maintaining documentation
* Enforcing project standards
* Assisting contributors
* Managing releases

Maintainers may approve and merge Pull Requests.

---

### Contributors

Contributors are individuals who submit:

* Pull Requests
* Documentation improvements
* Security reports
* Bug fixes
* Features
* Tests
* Design proposals

Contributors help shape the project through discussion and implementation.

---

## Decision Making

### Routine Decisions

Routine development decisions are handled through:

* Pull Requests
* Issue discussions
* Maintainer review

Examples:

* Bug fixes
* Documentation updates
* Refactoring
* Small features

---

### Major Decisions

Major decisions require maintainer review and may require Founder approval.

Examples:

* Architectural changes
* Breaking API changes
* New major dependencies
* Security-sensitive functionality
* Licensing changes

---

## Pull Request Process

A Pull Request may be merged when:

* CI checks pass
* Review feedback is addressed
* Project standards are met
* Required approvals are obtained

Maintainers may request changes before approval.

---

## Security Decisions

Security takes priority over feature development.

Maintainers may:

* Delay public disclosure of vulnerabilities
* Reject insecure implementations
* Require additional review for security-sensitive changes

Security-related decisions may be escalated directly to the Founder.

---

## Release Management

Official releases are created by project maintainers.

Release decisions include:

* Versioning
* Release schedules
* Changelogs
* Conversion records under the Sentinel Release-Based Licensing Policy

---

## Contributor Recognition

Sentinel recognizes meaningful community contributions.

Contributors may receive:

* Public acknowledgement
* Release note credits
* Community recognition
* Access to contributor programs
* Sentinel Enterprise credits
* Sentinel Cloud benefits

Rewards are granted solely at the discretion of The Deployment Lab.

Participation does not guarantee rewards or compensation.

---

## Maintainer Selection

Maintainers may be appointed based on:

* Technical expertise
* Consistent contributions
* Community engagement
* Security awareness
* Long-term commitment

Maintainer status may be revoked if necessary to protect the project.

---

## Licensing Authority

All licensing decisions, exceptions, commercial licensing arrangements, and future licensing changes remain under the authority of The Deployment Lab.

---

## Governance Changes

This governance document may be updated as the project evolves.

Changes become effective upon publication in the repository.
